# Product Roadmap

> Internal planning document. This stays out of the repository root on purpose.

## Product Shape

Magic Key Assistant is a local-first operations layer for solo operators and very small teams.

The durable product shape is:

- web console first
- Discord optional
- continuity by default
- accelerators opt-in
- records and review state treated as first-class product assets

## Core Workflows

The product succeeds when a new team can get real value from a small number of repeatable workflows.

The current anchor workflows are:

- meeting to actions and decisions
- lead to follow-up
- decision to recall and rationale
- continuity review of overdue, stale, unowned, or unresolved work

## Product Principles

- Trust before autonomy
- Durable records before ambient commentary
- Transferable setup before bespoke internal habits
- Local-first defaults before cloud convenience
- Boring reliability before feature sprawl

## Default Operating Model

The default experience should be useful without tuning.

That means:

- continuity-only mode is the baseline product
- scheduled sweeps surface real follow-through work
- background behavior should create or explain records
- setup should not require expert operator knowledge

## Opt-In Acceleration

When users want more, the system can expand into accelerator behavior.

The accelerator family currently centers on:

- Scout for research and discovery
- Dreamer for ideation and framing
- Rainmaker for pipeline follow-through
- Steward for health and operating rhythm
- Curator for corpus quality, structural analysis, and self-interrogation

These should remain optional. They are not the root product story.

## Near-Term Priorities

### Onboarding

- keep first run self-serve
- make local setup legible and recoverable
- seed enough example state that the product is understandable quickly

### Product Surface

- keep the web console coherent
- avoid multiplying pages without a strong reason
- prefer integrated workflows over isolated feature islands

### Data And Reliability

- preserve forward-only migration discipline
- keep review and audit trails attributable
- keep continuity calculations durable and visible

### Packaging

- maintain a straightforward local install path
- keep Docker and Windows packaging viable
- preserve simple evaluator flows

## Things To Demote Or Remove

These ideas should stay secondary unless they clearly improve adoption:

- persona theater as the main product story
- autonomous prose that does not change durable state
- hardcoded schedules, channels, or organization vocabulary
- feature surfaces that exist mainly because the code can support them

## Success Criteria

A release is moving in the right direction if new users can:

- install locally without developer hand-holding
- understand the operating model quickly
- capture real work into durable records
- see why the system is worth keeping open next week
